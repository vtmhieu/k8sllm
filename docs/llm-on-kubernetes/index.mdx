# LLM On Kubernetes

LLM workloads stress Kubernetes in unusual ways: GPU capacity is scarce, model startup is slow, latency has multiple phases, and request cost varies by token count. Treat LLM serving as a platform capability with its own SLOs.

![LLM inference stack on Kubernetes architecture](/img/architectures/llm-inference-stack.svg)

## Core capabilities

| Capability | Why it matters |
| --- | --- |
| GPU node pools | Keeps accelerator scheduling explicit and protects general workloads. |
| Model runtime | Handles batching, KV cache, streaming, and model loading. |
| Serving abstraction | Routes traffic, manages revisions, exposes autoscaling hooks. |
| RAG services | Adds retrieval, metadata policy, and evaluation loop. |
| Telemetry | Measures TTFT, throughput, queue wait, GPU saturation, and cost. |
